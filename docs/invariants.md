# opl-meta-agent 硬约束

Owner: `opl-meta-agent`
Purpose: `invariants_and_forbidden_claims`
State: `active_truth`
Machine boundary: 本文是人读硬约束。机器真相继续归 `contracts/`、source、CLI/API 行为、OPL Agent Lab result refs、work-order receipts、tests 和 target owner receipts / typed blockers。

- Do not store runtime artifacts in repo source.
- Do not implement generic OPL runtime primitives in this domain repo.
- Do not implement repo-owned generic CLI, MCP, Skill, product-entry, sidecar, status, or workbench wrappers; OPL Framework generates or hosts those surfaces from contracts.
- Do not treat OMA's `Foundry Agent OS` target delta as target-agent readiness, App live readiness, human approval, default promotion, or family production readiness. The target delta only fixes the split between OPL Agent OS, OMA agent-building authority kernel, and the OPL-owned capability registry ABI.
- Do not let Capability Registry, Agent Lab, Vault, Console, Runway, Pack, generated surfaces, work-order primitives, or App/workbench projections sign target owner receipts, create target-domain typed blockers, write target truth, write target artifact or memory bodies, authorize target quality/export verdicts, or promote a default target agent.
- Default operator/read-model interpretation must route through `current_owner_delta`; raw suite counts, generated interface readiness, registry discovery, conformance pass, work-order shape, or refs-only scaleout evidence cannot become the owner answer.
- Do not add target-domain-specific Agent Lab suite kinds, OMA command families, or top-level OMA surface kinds for MAS, MAG, RCA, or any single agent. Target agents expose standard handoff; Agent Lab and OMA consume the standard interface.
- Do not treat `stage_executor_policy_candidate` as an executor switch, runner implementation, queue implementation, Agent Lab replacement, quality equivalence claim, or default-executor promotion. OMA may propose refs-only stage executor policy candidates; OPL / Agent Lab owns trials, gates, read models, receipts, adapters, bindings, and any default-executor promotion decision.
- Non-default stage executor candidates require explicit adapter / executor binding / receipt refs and must fail closed with `blocker:missing_non_default_executor_binding_ref` when those refs are absent.
- Every generated target stage must declare progress-first `stage_completion_policy`: `closeout_packet_required=false`, `raw_artifact_sufficient_for_progress=true`, `next_stage_transition_owner=codex_cli`, and Framework route accept/reject authority false. Any readable artifact may advance; closeout/owner/reviewer refs control quality, delivery, promotion, export, publication, and ready claims only.
- Live Evidence deferred is a core OMA development rule. Script-to-pack hygiene, retained script active-caller classification, OPL primitive replacement mapping, generated / registry / App projection inputs, developer work-order and target-improvement policy contracts, no-forbidden-write guard, no-active-caller cleanup, tombstone/provenance, and no-resurrection guards are functional/structural lanes and may close before App live rendering, registry live discovery receipts, repeat target cohorts, real source patch -> rerun -> owner receipt loops, independent reviewer direct evidence, human approval, default promotion, or family production readiness. Those later items remain mandatory evidence lanes for target-agent readiness, default promotion, App live readiness, or production claims; controlled canary, suite pass, refs-only scaleout, OPL refs-only consumption, contract completeness, or generated interface readiness cannot replace them.
- Do not let OPL write domain truth, memory body, or quality/export verdicts.
- Keep `agent/` as a real domain pack: `agent/stages/manifest.json` is the only repo-tracked domain-owned OPL Pack stage source, and every policy, prompt, knowledge, quality-gate, action, and transition ref must resolve to a real repo file or canonical action. Hosted `family_stage_control_plane` is OPL-owned and must not be mirrored back into this repo.
- Keep Standard Agent identity language-neutral. OMA and every generated target Agent are defined by the Markdown/JSON declarative Pack; Python/TypeScript may appear only in declared helper implementations. Helper language, repo main language, package manager, source root, or native runtime must never become an Agent membership axis or a second Pack source.
- Every generated target implementation profile must keep generated surfaces owned by OPL, forbid repo-owned generic runtime/CLI/workbench, and declare only helper roots that physically exist and have passed target-repo review. A new baseline is pack-only; unimplemented helper needs belong to AgentPackPlan/capability requirements or developer work orders. OMA must not invent helpers or infer a language from keywords in the delivery domain.
- Keep Agent and OPL Package identity explicit: `pack_compiler_input.canonical_agent_id=oma` and package manifest `agent_id/package_id=oma`; repo slug, `domain_id`, npm package name, and Codex plugin carrier locator remain `opl-meta-agent` and must not be inferred as a second package identity.
- Generated interfaces may invoke declared actions and minimal authority functions, but they must remain projection/invocation surfaces owned by OPL Framework and must not escalate into domain truth, memory body, artifact body, quality/export verdict, or default-promotion authority.
- Do not train or deploy model weights from this repo.
- Do not promote a generated target agent as default without an explicit OPL gate.
- Do not write target-domain truth, memory body, artifact body, quality verdict, or export verdict.
- Mechanism patch proposals are proposal-only surfaces: they may reference editable mechanism surfaces, observed segment runs, evidence deltas, and next candidate refs, but they must not apply edits or promote defaults without an explicit gate.
- Generated-surface proof, registry readiness, App projection readiness, suite pass, schema completeness, or contract completeness must not be described as domain ready, App live rendering, quality verdict, owner receipt, or default promotion.
- A usable landing claim requires real consumption evidence: stage launch refs, independent Codex reviewer evaluation with direct evidence and no shared context, registry/App consumption receipt where applicable, real blocked target patch/rerun evidence, owner receipt, no-forbidden-write proof, and cleanup closeout.
- Default `build-agent-baseline` must consume OPL StageRun query readbacks for its ordered action route before materializing the target agent package. Each readback must bind the target domain, exact stage attempt, accepted typed closeout, canonical completed outcome, and conflict-free ledger identity. Domain closeout bodies are not copied into OPL runtime truth: the normalized packet must use canonical `domain_owned_stage_output_ref` v1 with matching `domain_id`, and its `output_ref` must also be a closeout ref. OMA accepts only `file://` output refs under the attempt `workspace_root`, requires matching SHA-256 metadata, rejects traversal/symlink escape, and rebinds payload stage/closeout identity to the ledger. `stage-decomposition` owns the machine-readable stage graph/action refs and file plan; `agent-skeleton-build` owns the matching file bodies. An incomplete route returns a typed continuation and emits no receipt.
- Do not encode target-agent stage sizing primarily in contracts, schemas, validators, generated files, or helper scripts. Stage size is an AI design judgment owned by the stage main prompt and, when needed, repo-local professional skills. A top-level stage should own one major open semantic judgment; deterministic generation, validation, file materialization, helper receipts, and readbacks stay inside that stage. Independent open judgments with different owners, knowledge sources, gates, handoffs, or failure routes must be split; purely mechanical wrappers around the same judgment must be merged or deleted.
- Do not turn mechanically repairable stage-decomposition admission issues into terminal blockers. Format, projection, ref, stage input, `requires`, and expected receipt drift that is fully derivable from valid design objects and authority refs must be repaired within the same stage admission/materialization path when possible, then revalidated. This progress-first rule must not synthesize missing semantic objects, missing source evidence, owner decisions, human gates, target-domain truth, owner receipts, or authority the target owner has not granted; those gaps remain route-back, typed blocker, or human gate.
- When the user supplies a paper/PDF/repo/product case as `reference_design_source`, source-derived design must treat that source as design origin and preserve content-rich ReferenceDesignPacket, TransferMap, AgentPackPlan, morphology, and DesignAdmissionReceipt refs. Research-driven design preserves the equivalent ResearchSynthesisPacket route. `StageDecompositionSubpacketSet` v2 stores them in unordered `design_object_packets`; validator compares stable identities/refs, not array position. Only explicit `dependency_edges` may require source evidence/safe packet before supported claims, design admission before materialization, and materialized bytes before `AgentBuildReceipt`. Transfer, pack planning, and morphology have no fixed cognitive order.
- `AgentBuildReceipt` must not be written as a pre-materialization expectation object. Before materialization, surfaces may carry only `expected_build_receipt_ref`; after materialization, the canonical receipt must include complete stage ids and OPL-computed file digests, be written once to `contracts/agent_build_receipt.json`, and be discoverable through the build action output path/ref. OMA consumes `opl-framework/standard-agent-pack-abi` through the OPL-managed Framework link and must not retain a local ABI body copy.
- `package.json` and `package-lock.json` must not declare or materialize `opl-framework` or its Temporal dependency closure. OPL owns Framework link install/update/sync/reinstall and repair; OMA verification may only perform the read-only OPL link check before commands and must fail closed without writing repo state.
- OPL external reference-design packets must remain refs-only and conform to `opl.reference_design_pattern_packet.v1`; OMA must not invent or accept a second content-rich external packet ABI. OMA may resolve only local JSON-pointer semantic refs within the packet directory, then own the content-rich `ReferenceDesignPacket`. Raw PDF/source refs、notes、opaque packet refs、missing pointers 或 missing steps/anchors 先作为 intake/extraction progress 保留，并阻止 source-derived design/scaffold/materialization ready claim，但不阻止 Codex 启动其他 declared stage；path traversal、不可读 bytes、invalid authority/non-claims、identity/currentness 或 source-material identity mismatch 才硬停。
- Every admitted reference workflow step must preserve source anchors and `source_derived|internal_synthesis` provenance. Seed anchors resolve through `anchor_catalog` with locator, selector, support role, verification, and version/fingerprint. Unresolved, mismatched, stale/unknown direct, or role-invalid anchors fail closed. User packet remains design origin; seed is fallback/secondary. Transfer mappings give every source step an explicit adopted/adapted/merged/stage-internal/rejected disposition; only distinct open judgments, owners, knowledge, gates, handoffs, or failure routes become independent target Stages, while all-open-judgments-in-one generic Stage is also invalid.
- Every declared reference source requires a matching typed packet source-material ref. When any user packet exists, seed patterns are context-only and must not expand the active TransferMap or stage graph. Every AgentPackPlan source workflow step must exist as a real stage-control-plane stage with matching pattern, step, anchors, prompt, skill, knowledge, tool-affordance, and quality-gate refs; plan-only and generic single-stage projections fail closed.
- Stage decomposition for a new target agent must produce an artifact morphology brief. The brief must cover native source format, artifact body owner, creative source refs versus assembled/export refs, sharding strategy, extent/scale contract, asset custody/file-path policy, thin assembler/helper boundary, and realistic target task review refs.
- Do not silently downgrade the target extent or scale declared by the user, owner, or source material. Scaffold, interface projection, sample output, demo defaults, suite pass, or script convenience cannot replace the declared chapters/pages/word count/assets/datasets/task scope/quality layer; missing evidence must become a blocker or owner route-back.
- Do not make Python/TS source strings the creative source of record for open-ended domain artifact bodies such as manuscripts, long-form reports, long decks, books, course material, narrative content, or other substantial authored deliverables. Helpers may assemble, validate, or export from native source files and manifests, but must stay thin and must not own artifact body authorship.
- Long or compound deliverables must have a domain-appropriate sharding strategy. Books, long reports, long decks, courses, datasets, asset packs, and similar deliverables require chapter/page/unit/dataset/asset shards or an equivalent owner-approved source structure rather than a single monolithic creative source.
- Imagegen outputs, external images, audio/video, datasets, templates, and other external assets require project-local reproducible custody: stable file paths, manifest/provenance/license refs, and owner boundary. Chat attachments, temporary URLs, local absolute paths, or untracked generated files are not completion evidence.
- Baseline, takeover, and external-suite reviewer evidence must cover artifact morphology risks with realistic target task evidence. Scaffold validation, generated interface projection, Agent Lab runner shape, or suite refs alone cannot prove artifact readiness or target-domain deliverable shape.
- `build-agent-baseline`, `takeover:test`, and `agent:evidence` may materialize AgentBuildReceipt, thin Foundry evaluation requests, Foundry evaluation work-order candidates, and proposal refs only. They must not compile or execute Agent Lab suites, write suite results, Foundry execution receipts, target owner receipts, learning/scaleout ledgers, promotion results, or target-domain typed-blocker bodies.
- Every Foundry evaluation work order must bind stable identity to `domain_id + target_agent_ref + descriptor_ref`, suite/task identity, and canonicalized source/reviewer/candidate provenance refs. The only active consumer aperture is `opl agent-lab evaluation-work-order execute --work-order <work-order.json> --output <dir>` owned by `one-person-lab/OPL Foundry Lab`; work-order ready is not evaluation complete or delivery complete.
- `improve:external-suite` must consume an explicit OPL Foundry Lab suite result via `--suite-result`. Missing execution receipt, quality floor, target-owned patch policy, or other declarative work-order inputs must return missing fields plus an expected typed-blocker ref without materializing a blocker body or executable work order.
- Free text stage decomposition, partial pack refs, missing independent gate policy, missing quality gate declaration, or self-review must fail closed before `baseline-delivery-receipt.json` is signed.
- The old direct script-authored stage graph and OMA-owned fixture/live stage runner must not remain as compatibility materialization paths. Tests may construct OPL ABI-shaped StageRun readbacks, but production intake is only `--stage-run-readback`; no local runner may launch attempts, synthesize receipts, produce an implicit graph, or act as a fallback route.
- An executable developer work order claim must include machine closeout refs for blocked suite, developer work order, patch traceability, target verification, target runtime/read-model consumption, workspace environment proof, no-forbidden-write, target owner receipt or typed blocker, patch absorption, worktree cleanup, and Agent Lab re-evaluation. Missing patch traceability, target verification, owner route, no-forbidden-write proof, or target owner receipt/typed-blocker return shape means OMA may emit only an expected blocker ref / route-back; the owning target or OPL surface materializes the blocker body.
- Executable developer work orders must also carry executor-first bundle refs: `executor_lease_ref`, `reviewer_pool_refs`, `patch_execution_bundle_ref`, and `target_closeout_refs`. Missing any of these refs, missing independent reviewer direct evidence/source refs, or missing rollback/canary/version refs must fail closed to an expected blocker ref. These refs do not authorize target-domain truth, memory body, artifact body, quality/export verdict, owner receipt authority, blocker-body authority, or default promotion.
- Executing an external developer work order must delegate to OPL Framework `work-order execute`. OMA must not implement or own target worktree lifecycle, generic runner, queue, attempt ledger, absorb, cleanup, post-absorb target owner closeout hook invocation, execution receipt, target truth, memory body, artifact body, owner receipt body, quality/export verdict, or default promotion.
- If a target-domain owner closeout hook is available, OMA may only record delegation proof. The hook owner is `target-domain via OPL`; OMA must keep `oma_can_write_owner_receipt=false` and must not synthesize owner receipt bodies from OPL execution results.
- Repo-local scripts must remain authority implementation refs, smoke helpers, fixture/proof helpers, or typed agent-building request builders. Generic file materialization, work-order execution delegation, receipt/lifecycle validation, and target worktree lifecycle must remain OPL-owned. Agent-building policy that grows beyond those roles must move back into `agent/`, `contracts/`, or explicit authority refs.
- Clean-room pattern sources such as `skillnerds/xskill` must not become runtime dependencies, daemon/server implementations, skill installers, generic schedulers, team servers, storage authorities, generated interface owners, or alternate execution planes in this repo.
- Xskill-derived trajectory atoms, candidate buffers, per-skill version / canary refs, and team redaction / sync refs may only guide refs-only proposals, work orders, blockers, reviewer evidence, or declarative agent-building policy. They cannot write target truth, memory body, artifact body, quality/export verdict, owner receipt body, default skill adoption, or promotion state.
- 正式 Stage Review 必须由新 StageAttempt 和新 executor session 完成；producer self-check、same-thread resume 或无 exact artifact hash / `no_context_inheritance=true` 的记录不能形成 Review receipt。
- 正式 baseline route 不得从 `baseline-run` 绕过 `optimizer-iteration` 进入 delivery；Agent Design Meta Review 只诊断并 route back，不内联修改上游 artifact。
