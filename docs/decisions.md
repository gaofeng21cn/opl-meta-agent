- Adopt OPL standard domain-agent scaffold v1.
- Provide OPL standard `user_stage_log_contract` on every agent-building stage. OMA closeout must explain the user-facing agent-building problem, stage goal, work done, changed source/contract/test/docs surfaces, outcome, remaining blockers, and evidence refs, or return a typed blocker. OPL only projects this into `stage_progress_log.user_stage_log` with duration/token/cost observability; it cannot infer agent-building semantics, write target domain truth, or authorize target quality/export/default promotion.
- Keep this repo as a declarative domain pack plus minimal authority functions.
- Treat `agent/` as the authoritative repo-tracked domain pack root. Pack file paths and stage prompt refs must be machine-checkable paths, not Markdown heading names or prose-only references.
- Use `opl-meta-agent` as the repo/product wording; use `meta-agent` only as a capability type.
- Keep `opl-meta-agent` outside the OPL Framework repo; OPL provides Agent Lab, scaffold surfaces, and generated interface bundles.
- Do not implement repo-local generic CLI/MCP/Skill/product-entry wrappers. OPL Framework derives those descriptors from standard action / stage contracts.
- Generated interface availability is accepted only with authority boundaries: OPL may generate CLI/MCP/Skill/product-entry/OpenAI/AI SDK descriptors from this repo's contracts and may invoke declared minimal authority functions, but generated surfaces remain OPL-owned and cannot claim target truth, memory, artifact, quality/export, or default-promotion authority.
- Prove the minimum self-learning loop with a generated fixture agent and OPL Agent Lab external suite before adding live domain delivery.
- Treat mechanism patch proposal as a proposal-only self-learning output beside gated candidates. It must carry observe/diagnose/edit refs and explicit authority flags, then wait for an OPL gate before any promoted mechanism change.
- Add an external-suite self-evolution path: when a domain owner projects a blocked Agent Lab suite, `opl-meta-agent` may consume the suite run, create a developer patch work order, and act as the developer that changes the target agent source repo. Source-level stage, skill, prompt, rubric, quality-contract, test, and documentation patches are allowed after the target-agent patch gate; target domain truth, memory body, artifacts, publication quality verdicts, and default agent promotion state remain forbidden.
- Developer patch work orders must include a gap-to-patch traceability matrix, target repo file hints, required test/receipt/prohibited-write evidence, and version-management controls. A work order that only lists broad editable refs is too weak for self-evolution because it leaves the source patch under-specified and shifts quality control back to the foreground developer.
- Developer patch work orders must also require read-only target runtime/read-model consumption verification. Passing source tests is insufficient when the blocked suite came from a live domain workflow; the closeout must show that the patched capability or route is visible in target runtime projections and no longer parks as a stale human handoff while target-owner work remains. This verification still cannot write target domain truth, artifacts, memory, quality verdicts, or submission readiness surfaces.
- When a target-agent patch depends on runtime extras, workspace profiles, or dependency locks, the developer patch work order must require read-only workspace-environment migration proof and owner-entry redrive proof. A clean target repo test is not enough unless the target workspace lock/profile and runtime owner entry consume the same environment without target-checkout `.venv` or generated metadata pollution.
- Developer patch work orders must carry executor-first aperture, independent reviewer provenance, patch traceability, target verification refs, owner route refs, no-forbidden-write proof, rollback/canary/version refs, target owner receipt or typed blocker return shape, and machine closeout refs. Missing reviewer direct evidence, source refs, predicted impact, target verification, owner route, no-forbidden-write proof, or closeout refs must fail closed to typed blocker rather than producing an executable patch order.
- Developer patch work orders must use OPL family Progress-First accounting. `target_progress_accounting.progress_delta_classification` and OPL generic `deliverable_progress_delta` / `platform_repair_delta` are required; OMA-specific `substantive_deliverable_delta_refs` and `platform_interface_repair_refs` are aliases only. Platform/interface repair, closeout plumbing, cleanup, absorption, Agent Lab re-evaluation, currentness repair, and refs-only ledger work cannot be counted as target-agent substantive deliverable progress.
- Developer patch work orders must carry `work_order_currentness.provider_owner_route_index_evidence`, binding target agent, Agent Lab eval result, work order and owner route to the current OPL route index / owner-route ledger / stage-attempt ledger packet. Missing ledger/route proof, stale tuple refs, or platform-only refs placed in deliverable progress must fail closed before delegation. Typed blockers and mechanism proposals must carry repeat budget, dead-letter/escalation refs, next allowed action, and next forced delta semantics so repeated blockers can escalate through OPL stall lineage instead of looping as fresh work.
- OMA must validate Progress-First accounting before delegating to OPL work-order execution. Shared deliverable/platform refs and OMA alias refs must match exactly; `progress_delta_classification` must match the ref counts; `typed_blocker` requires lineage, repeat budget, next allowed action and next forced delta; human gate or stop-loss cannot be counted as target-agent deliverable progress.
- OMA contracts, generated pack drafts and package metadata must pin the shared OPL Foundry-series Progress-First policy by release ref, version, commit ref and fingerprint, but must not copy the shared policy body into OMA as authority. `package.json` / `package-lock.json` carry the actual `opl-framework-shared` dependency pin so the contract-level `domain_dependency_pin_required` is machine-verifiable. OPL Framework remains the shared policy release owner; OMA remains the Agent Foundry domain adapter and keeps only domain-specific aliases, refs, work-order packets and typed blockers.
- Executable developer work order execution is delegated to OPL Framework via `opl work-order execute`. OMA may provide a thin validation/delegation entrypoint, but must not own a generic runner, target worktree lifecycle, queue, attempt ledger, absorb loop, cleanup loop, owner receipt body, target truth, target artifact body, or target quality/export verdict.
- Post-absorb target owner closeout is also delegated through OPL. When a target repo exposes a target-domain owner closeout hook, OPL may invoke it after absorb; OMA only records `owner_closeout_hook_delegated=true`, `target_owner_closeout_owner=target-domain via OPL`, and `oma_can_write_owner_receipt=false`.
- Make Codex `stage-decomposition` typed closeout the default input authority for `build-agent-baseline`. The generated target stage graph, action catalog, pack file bodies, independent gate policy, quality gate declarations, and no-forbidden-write policy must come from that closeout and must be reflected in receipts. The old direct script-authored graph has no compatibility path; any remaining `fixture` runner must consume an explicit proof/test typed closeout path for deterministic validation and cannot be described as a retained public entry, implicit graph producer, or fallback materialization route.

OMA is allowed to compile and materialize `artifact_native_contract_ref`, `stage_folder_contract`, `stage.json`, `attempt.json`, manifest, receipt, typed blocker, current pointer, canonical, export, lineage, retention, physical kernel locator, conformance and workbench consumption refs in `contracts/stage_control_plane.json`, generated target packs, Agent Lab takeover suites, developer work orders and typed blockers. OPL Framework owns stage folder lifecycle, physical kernel, runtime state, conformance, projection, recovery, Agent Lab runner, queue, attempt ledger, worktree lifecycle and promotion gate; OPL App / App contracts own workbench consumption; target domain owners own target truth, artifact body, quality/export verdict, owner receipt body and target owner closeout.
